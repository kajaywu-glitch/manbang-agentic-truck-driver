# 项目总设计方向：基于 Agentic AI 的卡车司机连续找货决策

## 1. 项目目标

本项目要实现一个可独立运行的司机找货智能体，在官方 `demo` 仿真环境中，针对每位司机从 `2026-03-01 00:00:00` 开始推进完整自然月经营决策。每一步只能选择 `take_order`、`wait`、`reposition` 三类动作之一，目标是在遵守赛题约束和司机偏好的前提下提升月度净收益。

核心不是单笔订单预测，而是事件驱动、多步连续经营决策。策略必须考虑当前位置、当前时间、货源有效期、装货时间窗、空驶成本、运输耗时、后续可观测机会和司机个性化规则。

## 2. 已准备数据与代码

- 官方压缩包已位于项目根目录：`demo_docs_release_20260509.zip`。
- 官方工程已解压到：`demo/`。
- 官方文档已解压到：`docs/`。
- 公开数据位于：`demo/server/data/cargo_dataset.jsonl` 与 `demo/server/data/drivers.json`。
- 本地数据规模：货源约 500000 条，司机 10 名；`cargo_dataset.jsonl` 解压后约 210MB。

注意：这些数据只能用于理解字段和本地评测。Agent 决策代码在仿真运行中禁止直接读取上述原始文件，也禁止复制后从其他路径直读。

## 3. 必须严格遵守的赛题要求

1. 决策入口保持官方接口：`ModelDecisionService.decide(driver_id) -> dict`。
2. 决策代码只能通过 `SimulationApiPort` 访问环境：
   - `get_driver_status(driver_id)`
   - `query_cargo(driver_id, latitude, longitude)`
   - `query_decision_history(driver_id, step)`
   - `model_chat_completion(payload)`
3. 禁止在决策中打开、解析、扫描或缓存 `cargo_dataset.jsonl`、`drivers.json` 及同内容文件。
4. 禁止按具体 `driver_id` 写死规则。偏好可以通过运行时返回的 `preferences` 文本解析成规则，但不能写 `if driver_id == "D009"` 这类逻辑。
5. 每步动作只允许：
   - `{"action": "take_order", "params": {"cargo_id": "..."} }`
   - `{"action": "wait", "params": {"duration_minutes": 正整数} }`
   - `{"action": "reposition", "params": {"latitude": 数值, "longitude": 数值} }`
6. `query_cargo` 会消耗仿真时间：默认最多返回 100 条，浏览批大小 10，因此返回 100 条会消耗 10 分钟。策略不能无限查询。
7. 正式评测会重新运行官方服务端和收益脚本，提交包不应包含 `demo/server/data/`。
8. 复赛约束：总仿真运行时长不超过 4 小时，每个司机 token 消耗不超过 500 万。

## 4. 官方仿真与评分口径

### 4.1 时间和距离

- 仿真时间单位为分钟。
- 仿真起点为 `2026-03-01 00:00:00`。
- 空驶与接单赴装货点距离使用 Haversine 大圆距离，单位 km。
- 空驶速度来自配置 `reposition_speed_km_per_hour`，demo 默认 60 km/h。
- 距离换时间：`ceil(distance_km / speed * 60)`，最少 1 分钟；但接单赴装货点距离接近 0 时为 0 分钟。

### 4.2 收益公式

每位司机：

```text
gross_income = 成功接单货源价格总和
distance_km = 空驶里程 + 成功接单干线里程
cost = distance_km * cost_per_km
net_income = gross_income - cost - preference_penalty
```

API 返回的候选货源中 `price` 已被官方 demo 转为元；收益脚本会用原始数据回查并按 `/100` 转元。

### 4.3 接单流程

接单动作会依次发生：

1. 从当前点空驶到货源 `start`。
2. 若早于 `load_time` 开始，则等待到开始时间。
3. 若晚于 `load_time` 结束，则接单失败且货源下架。
4. 运输 `cost_time_minutes` 分钟，完成后位置变为货源 `end`。
5. 若完单超过仿真月度上界，收益可能不计入。

## 5. 推荐实现架构

后续实现主要修改 `demo/agent/`，尽量不改 `demo/server/` 与 `demo/simkit/`。如必须增加辅助文件，放在 `demo/agent/` 内，并在 `demo/agent/README.md` 或 `demo/SUBMISSION.md` 说明。

建议模块：

- `model_decision_service.py`：保留官方入口，负责调用 Planner 并兜底输出合法动作。
- `planner.py`：核心滚动规划器，完成状态读取、候选生成、打分、动作选择。
- `preference_rules.py`：把 `preferences` 文本解析成可执行规则，并计算候选动作风险。
- `state_tracker.py`：从 `query_decision_history` 与本轮内部记忆重建司机累计状态。
- `geo.py`：Haversine、时间换算、区域/圆形禁入判断。
- `schemas.py`：内部候选动作与评分结果的数据结构。
- `llm_helper.py`：仅在必要时调用模型，例如偏好文本结构化和打分解释，不让模型直接随意输出动作。

整体采用“确定性规划为主，Qwen3.5-Flash 辅助理解偏好”的混合策略。这样更稳、更省 token，也更容易通过动作合法性校验；同时与复赛推荐基座模型保持一致。

## 6. 核心算法路线

### 6.1 滚动时域规划

每步决策使用 Model Predictive Control 思路：

1. 获取当前司机状态。
2. 查询当前位置附近候选货源。
3. 生成合法接单候选、休息候选、空驶候选。
4. 对每个候选估计本步净收益、时间占用、偏好风险和后续机会价值。
5. 选择综合评分最高且风险可控的动作。
6. 每步重新观察环境并重算，不依赖一次性全月静态计划。

### 6.2 接单候选评分

对每条货源计算：

```text
pickup_km = haversine(current, cargo.start)
pickup_minutes = 0 if pickup_km <= 1e-6 else ceil(pickup_km / speed * 60)
arrival_minutes = now + pickup_minutes
wait_for_load = max(0, load_start - arrival_minutes)
finish_minutes = max(arrival_minutes, load_start) + cost_time_minutes
haul_km = haversine(cargo.start, cargo.end)
gross = cargo.price
travel_cost = (pickup_km + haul_km) * driver.cost_per_km
base_net = gross - travel_cost
```

过滤规则：

- 货源车型不匹配则过滤。
- 到达时间晚于装货窗结束则过滤。
- 完单时间超过月度仿真上界则强降权或过滤。
- 命中高罚分或无封顶偏好风险则过滤。
- 接单动作起点或行驶区间落入“禁止接单/禁止空驶”的时间窗则过滤。
- 违反城市边界、禁入圆区、最大空驶、最大干线距离等规则则过滤。

评分建议：

```text
score =
  base_net
  + 0.25 * net_per_hour
  + destination_opportunity_bonus
  + preference_progress_bonus
  - waiting_penalty
  - deadline_risk_penalty
  - preference_violation_penalty
  - excessive_deadhead_penalty
```

其中 `net_per_hour = base_net / max(1, total_minutes / 60)`。权重先用可解释常量，后续根据本地评测结果小步调参。

### 6.3 后续机会价值

不允许读取全量货源文件，因此后续机会价值只能来自运行时观察与轻量先验：

- 记录每次 `query_cargo` 返回的候选目的地、起点城市、价格、有效时间。
- 维护按区域和小时段聚合的 `market_heat`：候选数、均价、平均净收益、平均等待时间。
- 对候选订单的终点，根据已观察 `market_heat` 给 `destination_opportunity_bonus`。
- 当观察不足时，使用保守策略：优先去已观测货源密度高、空驶成本低、且不冲突偏好的区域。

不要把公开数据统计硬编码进 Agent；公共数据和复赛隐藏数据可能不同。

### 6.4 休息策略

休息不是兜底等待，而是偏好管理和低谷避险工具。生成候选：

- `wait(30)`、`wait(60)`、`wait(120)`：普通低收益或无货场景。
- 等到下一个整点、早高峰、装货窗开始前的短休。
- 为满足连续休息偏好，生成 `wait(180/240/300/480)` 等规则需要的完整休息段。
- 夜间禁行或“禁止接单/空驶”窗口前，直接休息到窗口结束。
- 整天不出车偏好要在月初或低收益日主动安排，不要等月末被动补。

休息评分：

```text
wait_score =
  preference_progress_bonus
  + avoided_violation_penalty
  + expected_market_recovery
  - opportunity_cost
```

若所有接单候选风险高或净收益低，休息往往优于盲目空驶。

### 6.5 空驶策略

空驶只在“当前位置候选质量差且目标区域预期收益足以覆盖空驶成本”时使用。

候选目标来源：

- 当前查询结果中高分货源的装货点附近。
- 过去观察到高 `market_heat` 的区域中心。
- 司机偏好要求的家、目标点、接人点、允许边界内安全点。
- 避开禁入区或越界后的最近安全点。

可选的有限前瞻：

- 每若干小时最多做一次远端 `query_cargo` 探测，查询 1 到 3 个候选目标即可。
- 每次探测都会消耗仿真时间，必须把 `query_scan_cost_minutes` 计入评分。
- 如果当前附近已有高质量订单，不做额外探测。

空驶评分：

```text
reposition_score =
  expected_best_net_after_move
  - deadhead_km * cost_per_km
  - move_time_opportunity_cost
  - preference_violation_penalty
```

没有足够证据时，空驶评分应偏保守。

## 7. 偏好规则处理方案

偏好是得分关键，建议实现“文本解析 + 规则对象 + 动作模拟校验”。

### 7.1 解析方式

1. 先用正则识别高频规则：
   - 不接某货源品类。
   - 每日连续休息 N 小时。
   - 某时间段不接单、不空驶。
   - 自然月至少 N 个整天不接单或不空驶。
   - 单笔干线距离不得超过 N km。
   - 赴装货点空驶不得超过 N km。
   - 累计空驶不得超过 N km。
   - 每天接单不得超过 N 单。
   - 首单开工不得晚于指定时间。
   - 必须到达某坐标、必须在家、禁入圆区、市域边界。
   - 指定熟货源必须接。
2. 对无法识别的偏好，只调用一次 LLM 结构化，缓存结果。
3. 结构化结果必须保留原文、罚金额度、封顶信息和适用时间窗。

### 7.2 规则对象

每条规则实现两个方法：

```python
is_hard_constraint() -> bool
estimate_penalty(action_candidate, state) -> float
progress_bonus(action_candidate, state) -> float
```

高罚分、无封顶、长期连锁影响的规则视为硬约束，例如禁入区、夜间禁止出车、家事约定、指定熟货。低罚分且收益明显覆盖时可作为软约束，但要明确扣分。

### 7.3 状态重建

优先使用 `query_decision_history(driver_id, step)` 获取历史动作，结合当前状态维护：

- 每日接单数。
- 每日连续休息最长时长。
- 每日是否完全不出车。
- 夜间/午间禁行动作区间。
- 累计空驶里程。
- 已到达目标点的自然日集合。
- 家事/熟货等一次性任务完成状态。

不要通过读取 `demo/results/` 来恢复历史。

## 8. LLM 使用策略

基座模型明确选用 `qwen3.5-flash`。建议把 Qwen3.5-Flash 用在“偏好文本结构化、歧义解释、少量候选裁决”，不要把全量候选直接交给模型自由发挥。

推荐调用场景：

- 每个司机首次出现时，把 `preferences` 结构化成规则 JSON。
- 当确定性评分前 3 名分差很小，且偏好文本有歧义时，让模型做一次裁决。
- 异常兜底时，让模型在少量候选中选择合法动作。

工程约定：

- 所有模型请求都通过官方 `SimulationApiPort.model_chat_completion`。
- 请求体显式写入 `model: "qwen3.5-flash"`，避免误用其他模型。
- 没有 API key、模型超时或返回格式错误时，必须无损降级到确定性策略。
- 本地默认不开启模型调用；如需启用，使用 `AGENT_ENABLE_QWEN35_FLASH=1`。

不推荐：

- 每一步把 100 条货源原样塞进模型。
- 让模型输出未经本地校验的动作。
- 使用长链路推理文本；输出必须是 JSON。

最终动作返回前必须经过本地校验器，保证字段、类型和参数合法。

## 9. 兜底策略

任何异常都不能让仿真崩掉。建议兜底顺序：

1. 若存在合法且正收益明显的候选，选择最高分 `take_order`。
2. 若当前处于禁行动作窗口或需要补休，返回对应 `wait`。
3. 若有明确安全目标且空驶距离短，返回 `reposition`。
4. 否则返回 `wait(60)`；月底或临近窗口时缩短为 `wait(30)`。

兜底动作也要检查不违反硬规则。

- 保留无 key 降级：没有 API key 或模型失败时，完整仿真仍然必须跑通。
- 需要比较“不开模型确定性基线”和“启用 Qwen”的 31 天结果，记录收益、罚分、token 和耗时。

## 16. Hybrid Agent 切换门槛与实施计划

不要把确定性规则调参变成无限循环。当前项目已经满足进入 Hybrid Agent 的基础门槛，后续重点是验证和收紧 Qwen3.5-Flash 的受控参与，而不是回退到无限手写规则。

- 完整 31 天仿真可以跑完。
- 无 `validation_error`。
- 无仿真崩溃。
- 10 名司机动作日志非空。
- 无 API key 时确定性策略仍可独立运行。
- 主要罚分点已经可定位到规则类别和动作日志区间。
- 本地候选生成、动作校验、异常 fallback 已稳定。

不要求所有罚分清零后才进入 Hybrid。D009 home-night、D010 家事、连续休息、必访点等问题正是 Qwen3.5-Flash 应参与理解和复审的重点。

Hybrid Agent 第一版计划：

1. 冻结确定性底座：只修崩溃、非法动作、硬约束绕过，不再做大范围纯规则重构。
2. 接入偏好结构化：Qwen 输出 JSON hints，本地只允许新增或收紧约束。
3. 接入候选复审：本地先生成合法候选，Qwen 只返回候选 index。
4. 本地复核：模型选择必须通过本地校验，否则回退确定性首选。
5. 控制调用：用 `AGENT_QWEN_MAX_REVIEWS` 限制复审次数，优先 home-night、家事、连续休息、熟货、必访点和分数接近场景。
6. 对比评测：分别跑不开 Qwen 和启用 Qwen 的短测与 31 天完整评测，记录收益、罚分、token、耗时、模型调用次数。

Hybrid Agent 约束：

- Qwen3.5-Flash 是基座模型，但不是自由动作生成器。
- 模型调用只能通过 `SimulationApiPort.model_chat_completion`。
- 请求体显式使用 `model: "qwen3.5-flash"`。
- 不给模型原始全量数据文件。
- 不提交真实 API key、本地 config 或 results。
- 模型失败、超时、坏 JSON、候选越界时必须无损降级。
